## 3. mano-cua 执行问题与限制

### 3.1 工具能力边界

| 限制项 | 影响 | 规避方式 |
|--------|------|---------|
| 跨应用拖拽 | md2wechat L2.1 拖拽上传无法采集 | 人工录屏补充 |
| HTML date input | TripSplit 日期字段输入耗时极高（40+ 步） | DB 预填充 + GUI 验证 |
| DevTools 交互 | 控制台输入区域难以定位和操作 | 避免依赖 DevTools |

### 3.2 日期输入问题（详细）

Chrome 中文 locale 下的 HTML `<input type="date">` 字段（年/月/日格式）对 mano-cua 构成显著障碍：

- **问题现象**：`type` 命令无法直接写入日期字段；需退回到逐键 `key` 操作，每个日期耗费 15-20 步
- **根因**：date input 的三段式（年/月/日）各自独立，焦点不自动跳转，且 `type` 动作不触发浏览器原生 date picker 的段落切换
- **影响**：TripSplit L1.1 Golden 采集耗费 40 步（其中 25+ 步用于日期），L1.7 的第一次采集因此失败并被终止
- **规避措施**：对需要创建数据的用例，采用 SQLite 预填充 + mano-cua 仅做 GUI 验证的混合策略

### 3.3 执行效率

| 指标 | md2wechat | TripSplit |
|------|-----------|-----------|
| 平均步数/轨迹 | 10.3 | 9.8 |
| 单轨迹平均耗时 | ~5 min | ~4 min |
| 总采集耗时 | ~3.5h | ~2h |
| 一次性成功率 | ~70% | ~65% |

### 3.4 额外观察

- mano-cua 的 evaluation 模块（--expected-result）能自动判断任务成功/失败，confidence 值普遍为 1.0，判断准确
- stdout 需要 `PYTHONUNBUFFERED=1` 环境变量才能实时输出，否则全部缓冲到进程结束
- mano-cua 在简单验证任务（打开页面→点击→读取数据）上表现优秀，通常 3-5 步完成
- 涉及复杂表单填写（下拉框、多选框、日期选择器）时步数显著增加

## 4. Phase 0 流程评估

### 4.1 八步流程验证结论

Phase 0 定义的八步流程（PRD → 测试用例 → Golden → Buggy → Bug 清单 → 审核 → 采集 → 标注）在两个应用上均已完整跑通，验证了：

1. **流程可行性**：产品经理、研发经理、测试经理三角色分工明确，交接物清晰
2. **工具可用性**：mano-cua 作为 GUI 自动化采集工具，能覆盖 94%+ 的测试点
3. **质量保障**：6/6 Bug 全部被自动捕获，正负样本对质量可靠

### 4.2 两个应用对比

| 维度 | md2wechat | TripSplit |
|------|-----------|-----------|
| 应用类型 | 单页面静态渲染 | 多页面 CRUD + 计算 |
| 复杂度 | 低（纯前端） | 中（Flask + SQLite） |
| 表单复杂度 | 低（文件上传+文本框） | 高（日期、下拉、多选、比例输入） |
| mano-cua 适配度 | 高（94.7%覆盖） | 中（需 DB 预填充辅助） |
| Bug 类型 | 前端渲染/样式 | 后端逻辑/数据 |

## 5. 后续 Phase 扩展建议

### 5.1 短期优化（Phase 1 准备）

| 建议 | 优先级 | 说明 |
|------|--------|------|
| 标准化 DB seed 脚本 | 高 | 为每条测试用例预定义初始数据，避免手动 DB 操作 |
| 日期输入规避方案 | 高 | 应用层面提供 URL 参数预填或 API 接口创建数据 |
| mano-cua stdout 默认 unbuffered | 中 | 向 mano-cua 团队反馈，建议默认关闭输出缓冲 |
| 标注格式标准化 | 中 | 定义 JSON Schema，方便后续自动化处理和模型训练 |

### 5.2 中期扩展（Phase 1-2）

| 方向 | 说明 |
|------|------|
| 扩大应用池 | 增加更多类型的应用（移动端、桌面端、复杂 SPA） |
| 增加 Bug 类型 | 覆盖安全漏洞、性能退化、并发问题等 |
| 自动化 Pipeline | 将 8 步流程编排为 CI/CD pipeline，减少人工介入 |
| VLA 模型训练 | 利用已采集的 50 条轨迹 + 7 组正负样本对进行初步模型微调 |
| 跨浏览器测试 | 扩展到 Firefox/Safari，验证平台兼容性 |

### 5.3 长期方向

| 方向 | 说明 |
|------|------|
| 自动 Bug 定位 | 利用正负样本对训练模型，自动从 GUI 截图推断 Bug 类型和代码位置 |
| 自修复闭环 | 模型检测到 Bug → 生成修复建议 → 验证修复 → 回归测试 |
| 持续验收 | 每次代码提交自动触发 mano-cua 验收，替代部分人工 QA |

---

## 6. 总结

Phase 0 验证了 VLA 桌面自动化在软件测试验收中的可行性。两个不同类型的应用（前端渲染工具 + 后端 CRUD 应用）均实现了 100% 的 Bug 检出率，产出了 50 条有效轨迹和 7 组正负样本对。

主要瓶颈在于复杂表单交互（特别是浏览器原生 date picker）的自动化效率，可通过数据预填充和应用层 API 有效规避。整体流程已具备向 Phase 1 扩展的条件。
